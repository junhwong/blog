新的模板引擎，躁起来吧！

前端框架随着Javascript和互联网的发展是如雨后春笋一样不断的出现，如：ReactJS, AngularJS, VUE等等它们各有优缺点但最终目的是一样的，那就是操作浏览器/原生端元素(可以通俗的理解为 DOM ), 每个框架因为理念和技术(初期实现)的不同而造成再模板语法的不同，我们下面挑几个代表来看看，为了直观我尽可能找中文文档：

AngularJS2:

Angular 应用管理着用户之所见和所为，并通过 Component 类的实例（组件）和面向用户的模板来与用户交互。

从使用模型-视图-控制器 (MVC) 或模型-视图-视图模型 (MVVM) 的经验中，很多开发人员都熟悉了组件和模板这两个概念。 在 Angular 中，组件扮演着控制器或视图模型的角色，模板则扮演视图的角色。

VueJS:

Vue.js 使用了基于 HTML 的模版语法，允许开发者声明式地将 DOM 绑定至底层 Vue 实例的数据。所有 Vue.js 的模板都是合法的 HTML ，所以能被遵循规范的浏览器和 HTML 解析器解析。
在底层的实现上， Vue 将模板编译成虚拟 DOM 渲染函数。结合响应系统，在应用状态改变时， Vue 能够智能地计算出重新渲染组件的最小代价并应用到 DOM 操作上。
如果你熟悉虚拟 DOM 并且偏爱 JavaScript 的原始力量，你也可以不用模板，直接写渲染（render）函数，使用可选的 JSX 语法。官网.

我们将要做什么？

根据上面几种框架的模板语言可以得出我们的结论：

  • 类JSX语法： 语法不直观，上手难度大，但易于测试。
  • 类HTML语法：直观易上手，缺点是不易于测试。根据AngularJS2是后于 VueJS发布的，它也采用了这类语法，可见这类语法将更佳获得前端同学(特别是初学者)的青睐。

根据对比所以我们也将采用第二种语法来做并采用单文件作为单个组件(后期会详细评估)；虽然重复造轮子，但我们不会做再做一个框架(现在框架已经非常非常多了)，而只是做模板引擎并去适配这些流行的框架，让大家可以轻松的在这些框架上迁移(至少在模板这块)，这也同时希望和促进这些框架在本质和效率上有所提高并吧模板引擎这块分离使框架的体积变得更加的灵巧。

其它

XView 这个项目也将加入《Web全栈工程师实践》这个系列，XView 结过我多方面评估，将采用 Rust 语言来实现编译器：

  • 无 GC 安全，效率高；
  • 通用，理论上通过FFI可以在任何其它语言中调用（这是最主要的原因）；
  • 作为全栈工程师在有机会的时候不因该放过任何一个以用促学的实践机会。
